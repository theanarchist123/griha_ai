GRIHA AI — COMPLETE IMPLEMENTATION PLAN (REVISED)
Build an AI-powered property finding platform called Griha AI. The AI does all the hard work — searching listings, checking legals, negotiating with brokers, and reviewing contracts. The user just makes decisions.

NOTE ON TOOLS BEFORE STARTING
MongoDB Compass is a desktop GUI application for viewing MongoDB data. It is not a database host. The actual database lives on MongoDB Atlas free tier (M0, 512MB, permanently free). Connect Compass to your Atlas cluster using the connection string from Atlas — this gives you the familiar Compass interface locally while the live app uses the same Atlas database. Vercel hosts the Next.js frontend only. Render free tier hosts the FastAPI backend — it supports persistent processes, Playwright, and APScheduler, and deploys from GitHub exactly like Vercel.

IMPORTANT NOTE ON RENDER FREE TIER
Render free tier spins down after 15 minutes of inactivity. To prevent this add a self-ping mechanism. In main.py on startup schedule an APScheduler job that runs every 10 minutes and makes an HTTP GET request to the app's own /health endpoint. This keeps the service alive permanently at no cost.

------------------------------
AGENT 1 — SCRAPING AGENT
------------------------------

Tech used — Playwright for browser automation, Gemini Flash for fake detection, Gemini embedding-001 for embeddings, Motor for Atlas operations, APScheduler for scheduling.

The agent runs every 6 hours via APScheduler. It also runs immediately when a new user completes onboarding. It is manually triggerable via POST /api/internal/scrape with X-Internal-Key header.

The agent loads all active search profiles from Atlas. For each unique city, BHK, and budget combination it launches a scraping run. The same city and BHK combination is only scraped once per cycle even if multiple users need it.

Build a base scraper class with shared Playwright logic for launching headless Chromium, setting rotating user agent strings from a list of 5 real Chrome UA strings, navigating to a given URL and waiting for networkidle state, extracting data with CSS selectors, handling timeouts by skipping rather than crashing, and closing the browser cleanly. Add a random delay between 2 and 5 seconds between requests and simulate page scroll before extracting.

Build separate scraper classes for MagicBricks, 99acres, and NoBroker. Each constructs its search URL from city, BHK, and budget parameters and extracts title, price text, address, size in sqft, floor, furnishing status, amenities, images array, listing URL, and days listed from the listing cards.

After collecting raw listings run them through a processing pipeline. Normalize price strings to integers. Normalize size strings to integers. Run URL deduplication by checking source_url against Atlas. Skip any listing whose URL already exists. Run semantic deduplication for new listings by generating a temporary embedding and querying Atlas Vector Search with city and price range filters — skip if cosine similarity with top result exceeds 0.92. Run fake detection by sending listing data to Gemini Flash and asking for a JSON response with is_fake boolean, confidence score, and reasons array — mark as fake only if confidence exceeds 0.75. Generate the full embedding using Gemini embedding-001 on a text combining title, address, locality, BHK, furnishing, and amenities. Insert the property document into Atlas. After each batch completes trigger the matching agent and photo analysis agent as FastAPI BackgroundTasks with the array of new property IDs.

If a CAPTCHA is detected during scraping skip that run gracefully and log it. Do not crash.

------------------------------
AGENT 2 — MATCHING AGENT
------------------------------

Tech used — Gemini Flash for AI scoring, Motor for Atlas queries, Resend SDK for emails, APScheduler for scheduling.

The agent runs as a FastAPI BackgroundTask after scraping, when a user updates preferences, and as a full re-match daily at midnight via APScheduler.

Phase one is a hard MongoDB filter — query Atlas for properties matching city, price at or below budget_max, correct BHK, is_fake false, and not already rejected by this user. This reduces thousands of properties to around 50 candidates without any LLM calls.

Phase two sends each candidate to Gemini Flash for scoring. The prompt provides the full user preference profile and property details and asks for match score from 0 to 100, a breakdown object with booleans per preference, a single plain English AI insight sentence, and a recommended action. Store each result in the matches collection.

After each score check if score is 90 or above and immediately trigger an instant alert email via Resend.

Run the daily digest via APScheduler at 8 AM. Query matches from the last 24 hours per user. Take the top 5 by score. Send a branded HTML email via Resend with each property shown as a card. Mark included matches as digest_sent true in Atlas.

AGENT 3 — PHOTO ANALYSIS AGENT
------------------------------

Tech used — Gemini Vision free tier for defect detection, httpx for downloading image bytes, Motor for Atlas updates.

The agent runs as a FastAPI BackgroundTask after scraping. It also has a manual trigger at POST /api/internal/analyze-photos with property_id.

For each property load the images array from Atlas. Process a maximum of 6 images. For each image download the bytes using httpx and send directly to Gemini Vision. The prompt instructs detection of water damage signs, structural issues, poor maintenance, misleading photography, and incomplete construction. It explicitly instructs to only flag issues with clear visible evidence and to skip anything uncertain. Ask for a JSON response with has_issues boolean and issues array where each item has type, severity, confidence, and plain English description. Store only issues with confidence above 0.70. Update the property document in Atlas with the complete photo_red_flags array.

Generate the plain English summary using Gemini Flash with a prompt providing all check results and asking for a 3-sentence summary starting with the overall verdict. Store the complete report in Atlas. If risk is caution or high_risk send a legal alert email via Resend.

------------------------------
AGENT 5 — NEGOTIATION AGENT
------------------------------

Tech used — LangGraph for agent orchestration, Gemini Pro for message generation, Gemini Flash for broker response analysis, Twilio SDK for WhatsApp, Motor for Atlas state persistence.

The agent starts only on explicit user action via POST /api/negotiate/start with property_id, user_max_price, tone, and priorities. It never starts automatically.

Build the LangGraph state as a TypedDict with property, user_prefs, market_data, messages, current_offer, broker_last_response, turn_count, status, and analysis fields.

Node 1 — research_market — queries Atlas for comparable properties in the same locality and BHK listed in the last 90 days with price within 20 percent of target. Computes average, minimum, and maximum price using MongoDB aggregation. No LLM call.

Node 2 — craft_opening_message — calls Gemini Pro with market data, listed versus fair value gap, days on market, photo red flags as leverage, and user tone. Calculates opening offer as 87 percent of listed price. Waits 2 minutes using asyncio sleep before sending to simulate human behavior.

Node 3 — send_whatsapp_message — calls Twilio SDK directly to send the message to the broker's WhatsApp number. Appends negotiation_id as a query parameter to the Twilio inbound webhook URL. Stores the message in Atlas. Saves complete LangGraph state to Atlas. Sets status to waiting_for_broker and stops.

The graph does not poll for responses. When the broker replies Twilio fires a webhook to POST /api/internal/negotiation-response. FastAPI receives this, reads From number and Body from the Twilio form-encoded payload, matches to the active negotiation in Atlas using the From number or the negotiation_id query parameter, loads the saved LangGraph state from Atlas, and resumes the graph.

Node 4 — analyze_broker_response — sends broker message to Gemini Flash asking for price offered if any, broker sentiment as willing or firm or hostile, and recommended next action as counter, agree, walk_away, or escalate_to_user.

The routing function checks turn_count first. If turn_count exceeds 6 route to escalate_to_user. Otherwise route based on recommended action.

Node 5 — craft_counter_offer — calculates next offer with a step size that decreases as the offer approaches user maximum. Never exceeds user maximum. Calls Gemini Pro to write the counter message in the chosen tone.

Node 6 — close_deal — sends confirmation to broker, updates negotiation status to closed_won in Atlas, sends congratulations email to user via Resend.

Node 7 — escalate_to_user — sends a WhatsApp message to the user via Twilio explaining the situation and asking them to reply 1 to accept, 2 to make a final offer, or 3 to walk away. Saves state and pauses. Resumes when the user replies via the same Twilio inbound webhook.

------------------------------
AGENT 6 — CONTRACT AGENT
------------------------------

Tech used — OCR.space free API for text extraction, PyMuPDF for initial PDF processing, Gemini Flash for clause splitting, LangChain with Atlas Vector Search for RAG clause analysis, Gemini Pro for redline generation, Cloudinary for output storage.

The agent runs as a FastAPI BackgroundTask when a document of type rent_agreement or sale_deed is uploaded.

First extract text from the uploaded document. Use PyMuPDF to attempt direct text extraction. If extracted text exceeds 200 characters it is a text-based PDF — use that text. If less than 200 characters it is a scanned PDF — send the file bytes to the OCR.space API at api.ocr.space/parse/image with language set to eng, isTable set to false, and detectOrientation set to true. Extract parsed_text from the response and store it in the document's extracted_text field in Atlas.

Next split the document into clauses. Send the full extracted text to Gemini Flash asking it to identify and separate each clause and return a JSON array where each item has clause_number, heading if any, and verbatim clause text.

Analyze each clause using LangChain RAG. For each clause run a vector similarity search against the legal_corpus collection in Atlas to retrieve the top 3 most relevant sections of Indian law. Build a LangChain stuff_documents_chain that takes the retrieved law sections and clause text and calls Gemini Pro to determine risk level as standard, caution, or high, plain English meaning, specific problem if risky, recommended corrected wording, and relevant law section. Store complete analysis in the clause_analysis array in Atlas.

Generate the redlined counter-draft. Filter clauses where risk is high. Reconstruct the document replacing high-risk clause text with the recommended corrected wording. Send the reconstructed text to Gemini Pro to format as a clean professional agreement. Upload to Cloudinary. Store the URL as redline_url in Atlas.

For document Q&A at POST /api/documents/ask retrieve all text chunks belonging to the user from Atlas filtered by user_id. Build a LangChain ConversationalRetrievalChain using Atlas Vector Search as the retriever and Gemini Pro as the LLM. Return the answer and the source clause reference.

Before launch run the ingest_legal_corpus.py script once. This loads the Indian Contract Act 1872, Model Tenancy Act 2021, Maharashtra Rent Control Act 1999, Transfer of Property Act 1882 relevant sections, and RERA Act 2016 relevant sections. Split each into 500-token chunks with 50-token overlap. Generate Gemini embeddings. Store all chunks in the legal_corpus collection in Atlas with source act name and section reference as metadata.

------------------------------
AGENT 7 — NEIGHBOURHOOD AGENT
------------------------------

Tech used — Google Maps Distance Matrix API for commute times, Google Places API for amenities, Playwright for price trend scraping, data.gov.in API for AQI data, LangChain with Atlas Vector Search for sentiment RAG, Gemini Flash for sentiment synthesis, APScheduler for weekly refresh.

The agent runs when a user opens a property detail page and no neighbourhood report exists for that locality, and as an APScheduler job every Sunday at 3 AM to refresh cached reports. Reports are cached per locality not per property. Always check expires_at first — if the cached report is still fresh return it immediately without re-running.

For commute data call Google Maps Distance Matrix API with the property coordinates as origin and the user's commute destination from their search profile as destination. Request results for all four travel modes. Use departure time set to next Monday at 9 AM for peak hour and 11 AM for off-peak.

For amenities call Google Places API nearby search with a 1500 meter radius around the property. Run separate queries for each category. Take the top 3 results per category sorted by rating.

For flood risk scrape the IMD website for historical waterlogging records for the locality's PIN code. Classify as Low, Medium, or High based on incident frequency over the last 5 years.

For AQI call the data.gov.in CPCB air quality public API and filter by the nearest monitoring station to the property's city.

For noise level compute programmatically from the property's distance to major highways using Google Maps Roads API, distance to railway lines, and distance to the nearest airport. Within 200 meters of a highway is High. Between 200 and 800 meters is Medium. Beyond 800 meters is Low.

For price trends use Playwright to scrape the locality price trends section on MagicBricks for the specific locality and BHK type. Extract 12 months of average rental prices.

For resident sentiment scrape Google Places reviews for the top 5 housing societies in the locality and locality review pages on Housing.com and MagicBricks. Store all raw review texts in the neighbourhood_reviews collection in Atlas with embeddings. Build a LangChain map_reduce chain that retrieves all stored reviews for the locality, extracts positives and negatives from each, and reduces to top 3 positives, top 3 concerns, and an overall sentiment score.

Compute livability score in Python without any LLM. Score connectivity from metro distance. Score amenities from count of categories with results within 1 kilometer. Score environment by inverting AQI. Score affordability by comparing area average rent to city average. Score safety from proximity to police station plus civic complaint density. Compute overall as the mean of the five sub-scores.

Store the complete report in Atlas with expires_at set to 7 days later.

INTERNAL ENDPOINTS
------------------------------

Create an internal.py router with prefix /api/internal. All routes are protected by X-Internal-Key header check, not Clerk auth.

AUTHENTICATION
------------------------------

Create auth_middleware.py with a verify_clerk_token dependency that reads the Bearer token from the Authorization header, calls the Clerk API to verify it, and returns the decoded user payload. Use this as a FastAPI Depends on all user-facing routes.

On user first login call POST /api/auth/sync-user to create the user document in Atlas if it does not already exist.

DEPLOYMENT
------------------------------

Frontend deployment on Vercel. Connect your GitHub repo. Set the root directory to the frontend folder. Add all NEXT_PUBLIC environment variables in Vercel project settings. Deploy.

Backend deployment on Render. Connect your GitHub repo. Set the root directory to the backend folder. Set the build command to pip install -r requirements.txt and playwright install chromium. Set the start command to uvicorn main:app --host 0.0.0.0 --port $PORT. Add all backend environment variables in Render service settings. The self-ping APScheduler job runs every 10 minutes to keep the free tier alive. Add a GET /health endpoint in main.py that returns status ok — Render uses this to confirm the service is running.

MongoDB Compass setup for local development. Open Compass. Click New Connection. Paste the Atlas connection string from the Atlas dashboard. Click Connect. You now have full GUI access to all your Atlas collections during development.
